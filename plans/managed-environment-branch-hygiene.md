# Managed Environment Branch Hygiene

> **Status:** Preserved plan artifact from thread `thr_ifyh5ev5h8`.
> Review and update this plan before using it for implementation work.

## Goal

Prevent BB-managed environments from leaving stale local git branches behind
after their worktrees are cleaned up, without deleting user-owned work or
surprising users who intentionally keep branches around.

This plan covers local branches only. Remote branch cleanup is out of scope
unless Michael explicitly opts into it later.

## TL;DR / Reading Map

- Branch deletion is allowed only for persisted BB-owned managed worktree
  branches with fresh heads, no checkout blockers, and safe merge proof unless
  the user explicitly requests force branch cleanup.
- `environment_branch_refs` records branch ownership and identity only.
  Durable cleanup lifecycle and results live on `environment_operations`.
- The daemon receives source git common dir, branch, expected SHA, and explicit
  policy inputs; it validates only host-local git facts under lock.
- Provisioning must stop `git worktree add -B` from resetting unknown or
  user-owned branches.
- Existing polluted branches are audit-first and backfilled as `unknown`, not
  automatically deleted.
- Read the plan in this order: Branch Ownership Policy, Data Model Plan,
  Canonical Wire Contract, Server And Daemon Ownership, Async Lifecycle Wiring,
  Migration Strategy, Implementation Phases, then Validation Plan.

## Glossary

- Workspace cleanup: host-local removal of the managed worktree or managed
  clone directory.
- Environment cleanup: the server-owned lifecycle that decides when a managed
  environment can be destroyed and queues `environment.destroy`.
- Branch cleanup: deletion, dry-run, or audit of a local git branch ref after
  workspace cleanup.
- Managed environment: an environment with `managed=true`.
- BB-owned branch: a branch with persisted ownership `bb_created` in
  `environment_branch_refs`.
- User-owned branch: a branch with persisted ownership `user_owned`, or a branch
  that has no BB ownership record.
- Unknown ownership: persisted ownership `unknown`. Existing/backfilled
  branches default here and are not eligible for automatic deletion.
- Safe mode: branch cleanup mode that deletes only when BB can prove ownership,
  freshness, protection, checkout state, and effective-merge safety.
- Force mode: explicit branch cleanup mode that skips effective-merge proof but
  still enforces BB ownership, freshness, protected-branch guards, and
  checked-out-worktree guards.
- Force archive: an archive/delete/project-delete flow that is allowed to remove
  the workspace even when the workspace has risky work.
- Force branch cleanup: an explicit user action that allows deleting an
  otherwise unmerged BB-owned local branch. It is separate from force archive.
- Effectively merged: the branch has no remaining net contribution relative to
  the chosen base. This includes ordinary merge/cherry-pick equivalence and
  squash-merge detection, but it does not prove every original commit landed.

## Current Lifecycle

Creation:

- Thread provisioning creates an environment record in
  `apps/server/src/services/threads/thread-provisioning-environment.ts`.
- Managed environment branches are generated by
  `buildManagedBranchName()` in
  `apps/server/src/services/threads/thread-create-helpers.ts` as
  `bb/<threadId>` or `bb/<slug>-<threadId>`.
- Managed worktrees are placed under
  `<dataDir>/worktrees/<environmentId>/<repo-dir>` by
  `resolveManagedTargetPath()` in
  `apps/server/src/services/threads/worktree-paths.ts`.
- The daemon provisions a managed worktree with
  `git worktree add -B <branchName> <targetPath>` in
  `packages/host-workspace/src/provisioning.ts`. This creates or resets a
  local branch in the source repository's shared refs.
- Managed clones use `git clone` plus `git checkout -B <branchName>` inside the
  clone. Deleting the clone directory removes that clone-local branch, so the
  branch pollution problem is primarily managed worktrees.
- Manual unmanaged environments are validated only. They are not created or
  owned by BB, even if they are git worktrees.

Cleanup and archive:

- Archiving a thread calls `validateEnvironmentCleanupRequest()`,
  `requestEnvironmentCleanup()`, and `advanceEnvironmentCleanup()` in
  `apps/server/src/services/environments/environment-cleanup.ts`.
- Cleanup is requested only when a managed environment has no other live
  non-archived, non-deleted threads.
- Safe cleanup checks workspace status and rejects with
  `archive_confirmation_required` if the workspace has uncommitted changes or
  committed-unmerged changes relative to the merge base.
- Force cleanup bypasses that safety check.
- The server queues `environment.destroy`; the daemon calls
  `RuntimeManager.destroyEnvironment()`, which removes the worktree or clone.
- `removeWorktree()` removes the worktree path but does not delete the local
  branch ref.

Delete and project deletion:

- Thread deletion requests force cleanup and advances the same environment
  destroy lifecycle.
- Project deletion force-cleans managed environments before deleting the
  project.
- Both paths currently preserve local branches.

Squash merge:

- `workspace.squash_merge` creates a temporary detached worktree, squash merges
  the environment branch into the target branch, publishes the resulting commit,
  and removes the temporary worktree.
- Squash merge does not clean up the environment worktree and does not delete
  the environment branch.
- `Workspace.getStatus()` already includes logic to treat squash-merged
  branches as effectively merged for status display.

## Branch Ownership Policy

Automatically deletable branches must satisfy all of these conditions:

- The environment is `managed=true`.
- The environment `workspaceProvisionType` is `managed-worktree`.
- The branch has a persisted branch ref row with ownership `bb_created`.
- The branch name equals the tracked environment branch name.
- The branch is not the default branch, merge-base branch, current primary
  checkout branch, or any configured protected branch.
- The branch is not checked out in any git worktree.
- No live environment or thread still references the same owned branch.
- The branch head matches the expected head SHA at the moment of deletion.
- In safe mode, the branch is effectively merged into the selected base.

Branches that must be preserved unless the user explicitly selects them in a
manual cleanup command:

- Branches from unmanaged environments.
- Branches from manual unmanaged worktrees.
- Branches with ownership `unknown`.
- Branches with ownership `user_owned`.
- Branches not matching the stored environment branch.
- Branches created before the ownership model exists and only guessed from a
  name pattern.
- Branches with unmerged commits.
- Branches checked out in another worktree.
- Protected/default branches.
- Remote branches.

Force archive should continue to remove the managed workspace, but it should not
automatically delete a branch with unmerged commits. The branch is the user's
last durable ref after the worktree is gone. Preserve it and report that branch
cleanup is blocked until the user explicitly runs force branch cleanup.

## Effective Merge Semantics

Destructive safe deletion gates on "no remaining net contribution", not on
"every original branch commit landed".

Safe deletion may proceed when the daemon can prove one of these:

- The branch is not ahead of the merge-base branch after cherry-pick
  equivalence filtering.
- The branch's cumulative patch is present on the base as a squash-equivalent
  commit.
- The branch's cumulative diff against the base is empty.

Caveats to document in the implementation and user-facing copy:

- An empty cumulative diff means the branch contributes no net changes. It does
  not mean every commit was merged.
- Squash detection is bounded. If the scan cap is reached or proof is
  inconclusive, safe deletion must block with `not_effectively_merged`.
- Use a named implementation constant such as
  `BRANCH_CLEANUP_SQUASH_SCAN_COMMIT_LIMIT = 1000` for the initial squash
  equivalence scan cap. The cap applies per branch evaluation; project/global
  audits should additionally batch repositories and limit concurrent daemon
  scans.
- Effective merge detection is a destructive-deletion safety check, so fallback
  behavior must be pessimistic.

## Recommended Behavior

Use automatic deletion only for clearly safe BB-owned branches, and expose an
explicit branch cleanup action for everything else.

1. On successful managed worktree provisioning, record that BB owns the
   generated branch.
2. On safe environment cleanup, queue branch deletion after the worktree destroy
   succeeds when eligibility is clean.
3. On force environment cleanup, destroy the workspace but mark branch cleanup
   blocked when unmerged commits, dirty work, missing merge-base data, or
   ownership uncertainty exists.
4. On squash merge, do not delete the branch immediately. The environment is
   still using it. Instead, let later cleanup delete it if the branch is
   effectively merged. The user must still archive the environment or run an
   explicit cleanup flow after squash merge; squash merge itself should not
   auto-archive or delete the active branch.
5. Provide a dry-run audit command and an explicit apply command for existing
   polluted branches and force cases.

## Provisioning Branch Collision Policy

Managed worktree provisioning currently uses `git worktree add -B`, which can
reset an existing local branch. Branch hygiene must not make that more
dangerous.

Before invoking `git worktree add -B <branchName>`, BB must prove the target
branch is safe to create or reuse:

- If the local branch does not exist, provisioning may create it.
- If the local branch exists and is BB-owned by the same environment for a
  reprovision, provisioning may reuse/reset it only after validating the current
  head and ownership.
- If the local branch exists with ownership `unknown`, ownership `user_owned`,
  or no active branch ref row, provisioning must not silently reset it. The
  server should either fail provisioning with a clear collision error or choose
  a new generated branch name.
- Collision checks must run again in the daemon immediately before branch
  creation/reuse, under the same git metadata lock used for worktree mutations,
  so concurrent provisioning cannot race into branch reset.

This policy is separate from branch deletion cleanup, but it is required before
implementing automatic cleanup because both features depend on trustworthy
branch ownership.

## Data Model Plan

Add explicit branch ownership state instead of inferring ownership from branch
names during cleanup.

Recommended model:

- Add an `environment_branch_refs` table with:
  - `id`: generated primary key.
  - `environmentId`: required FK to `environments.id` with
    `ON DELETE CASCADE`.
  - `hostId`: denormalized from the environment for indexed host lookups.
    Treat the environment host as immutable after provisioning for this
    feature. The environment row remains authoritative; writers validate the
    value at branch-ref creation, and future host-migration work must either
    create a new branch ref or update branch refs in the same transaction as
    the environment host change.
  - `sourceGitCommonDir`: absolute git common dir for the source repository.
  - `branchName`.
  - `ownership`: `bb_created`, `user_owned`, or `unknown`.
  - `initialHeadSha`.
  - `lastObservedHeadSha`.
  - `deletedAt`.
- Add indexes for `environmentId`, `hostId`, and
  `(hostId, sourceGitCommonDir, branchName)`. Do not use
  `(hostId, sourcePath, branchName)` as a natural key or identity.
- Treat `environment_branch_refs` as per-environment ownership claims, not as
  canonical branch identity rows. Multiple active branch refs may point at the
  same `(hostId, sourceGitCommonDir, branchName)` when multiple environments
  share one local branch. That is what the `shared_owner` blocker represents.
- Enforce at most one active branch ref per environment with a partial unique
  index such as
  `UNIQUE(environmentId) WHERE deletedAt IS NULL`. Historical rows are valid
  only after the older row has `deletedAt` set; set `deletedAt` before inserting
  a replacement row for the same environment.
- Keep `(hostId, sourceGitCommonDir, branchName)` as a non-unique lookup index.
  Cleanup and audit use it to find active sibling claims and block deletion
  with `shared_owner` until all active claims are destroyed or deleted.
- Add an `environment_operations` kind for branch deletion, for example
  `delete_branch`.
- Keep branch deletion lifecycle separate from `environment.status`. Branch
  deletion is not another environment status state.
- Store cleanup eligibility and action inputs in operation payloads, not in
  generic metadata helpers.
- Store durable cleanup lifecycle state and daemon result payloads on
  `environment_operations`, not on `environment_branch_refs`. Branch refs
  represent ownership and branch identity only.
- API/CLI/UI read models may expose a derived `BranchCleanupReport` assembled
  from the branch ref and the active or most recent `delete_branch` operation.
  Do not persist branch cleanup queue states, result codes, or blocker codes on
  the branch ref row.
- `deletedAt` on `environment_branch_refs` is a denormalized cleanup marker.
  The terminal apply operation result is the source of truth; when a delete or
  already-missing operation completes, update `branchRef.deletedAt` in the same
  transaction to equal that operation's completion time.

Persisted ownership enum:

- `bb_created`: BB created the local branch for this managed worktree.
- `user_owned`: the user explicitly marked the branch as not BB-owned.
- `unknown`: ownership is not proven. Existing/backfilled rows use this value.

Do not persist a fourth "probable" ownership value unless it is intentionally
added as a first-class enum with full review. Migration/audit may compute a
non-persisted confidence field such as `auditConfidence: "bb_pattern_match"`
for display, but that confidence must not make a branch automatically
deletable.

Ownership review and claim path:

- Backfilled `unknown` rows are preserved by default, but they are not a
  permanent dead end. Add an explicit audit/review action that can promote an
  `unknown` row to `bb_created` only when the user selects the environment or
  branch, the environment is a managed worktree, the stored branch name and
  source git common dir match current daemon observations, the current head
  matches the reviewed `expectedHeadSha`, and no conflicting user-owned claim
  exists.
- Promotion from `unknown` to `bb_created` does not delete the branch in the
  same request. A later cleanup request must re-run normal safe/force
  eligibility checks.
- Repair sweeps must not downgrade an existing `bb_created` row to `unknown`
  because of transient proof failures such as disconnected hosts, temporary
  source unavailability, or git lock contention. Record a blocker or retryable
  failure instead. Create `unknown` only for newly reconstructed rows whose
  ownership cannot be proven.

Create branch ref rows after successful provisioning, not when provisioning is
requested. A failed provisioning flow may roll back the worktree or leave no
branch to own, so request-time ownership would be false. The provisioning
success path should record:

- branch name
- initial head SHA
- source git common dir
- host id and environment id
- ownership `bb_created` only for managed worktrees

The provisioning success handler should write the branch ref in the same DB
transaction that marks the environment `ready` when possible. If that write
cannot complete, the recovery behavior is owned by the Branch Ref Recovery And
Reconciliation section below.

## Canonical Wire Contract

Define one environment branch cleanup report shape plus separate daemon result
shapes for dry-run and apply. Use these everywhere: environment responses,
archive reporting, CLI JSON, UI display, and daemon result handling. Do not
maintain parallel durable enums for daemon result, archive outcome, display
status, and branch-ref state.

`BranchCleanupReport` should be a discriminated union assembled by the server:

- `{ kind: "not_applicable" }`: the environment has no local branch cleanup
  target, such as unmanaged environments or managed clones.
- `{ kind: "not_tracked", environmentId, reason: "missing_branch_ref" }`: the
  environment is a managed worktree that may have a local branch, but BB has no
  branch ref row yet. This is audit/backfill-needed, not proof that cleanup is
  unnecessary.
- `{ kind: "not_attempted", branchRefId }`: a branch ref exists, but there is
  no active or completed `delete_branch` operation for it.
- `{ kind: "pending", branchRefId, operationId, operationState }`: a
  `delete_branch` environment operation exists and is not terminal. The
  operation state comes from `environment_operations`; branch refs never store
  `requested`, `queued`, or similar lifecycle states.
- `{ kind: "dry_run_result", branchRefId, result }`: a dry-run request returned
  a `BranchCleanupDryRunResult` without creating durable lifecycle state.
- `{ kind: "operation_result", branchRefId, operationId, result }`: the most
  recent relevant `delete_branch` operation is terminal and returned a
  `BranchCleanupApplyResult`.

`BranchCleanupDryRunResult`:

- `{ code: "eligible", expectedHeadSha }`: dry run proved safe deletion at the
  observed head, but no ref was deleted.
- `{ code: "blocked", blockerCode, expectedHeadSha?, message? }`
- `{ code: "failed", blockerCode, message? }`

`BranchCleanupApplyResult`:

- `{ code: "deleted", deletedAt }`
- `{ code: "already_missing", deletedAt }`
- `{ code: "blocked", blockerCode, message? }`
- `{ code: "failed", blockerCode, message? }`

`eligible` is not a valid apply or operation result. The operation result
schema must reject it so apply cannot persist a "would be safe" outcome as if
it were a completed cleanup.

`BranchCleanupBlockerCode`:

- `not_bb_owned`
- `unknown_ownership`
- `protected_branch`
- `checked_out_elsewhere`
- `head_changed`
- `not_effectively_merged`
- `missing_merge_base`
- `shared_owner`
- `source_repo_unavailable`
- `host_disconnected`
- `git_error`

Daemon dry-run command results should use `BranchCleanupDryRunResult`. Daemon
apply command results and `environment_operations` terminal payloads should use
`BranchCleanupApplyResult`.
Server responses should return the full `BranchCleanupReport`. Archive
outcomes and display statuses are derived fields in response builders, not
independent persisted state.

Derivation mapping:

| Server report                                                       | UI status        | Archive outcome   |
| ------------------------------------------------------------------- | ---------------- | ----------------- |
| `{ kind: "not_applicable" }`                                        | `not_applicable` | `not_applicable`  |
| `{ kind: "not_tracked" }`                                           | `audit_needed`   | `skipped`         |
| `{ kind: "not_attempted" }`                                         | `not_attempted`  | `skipped`         |
| `{ kind: "pending" }`                                               | `pending`        | `pending`         |
| `{ kind: "dry_run_result", result: { code: "eligible" } }`          | `eligible`       | `skipped`         |
| `{ kind: "dry_run_result", result: { code: "blocked" } }`           | `blocked`        | `blocked`         |
| `{ kind: "dry_run_result", result: { code: "failed" } }`            | `failed`         | `failed`          |
| `{ kind: "operation_result", result: { code: "deleted" } }`         | `deleted`        | `deleted`         |
| `{ kind: "operation_result", result: { code: "already_missing" } }` | `deleted`        | `already_missing` |
| `{ kind: "operation_result", result: { code: "blocked" } }`         | `blocked`        | `blocked`         |
| `{ kind: "operation_result", result: { code: "failed" } }`          | `failed`         | `failed`          |

Human text is derived from `kind`, `result.code`, and `blockerCode`. Optional
diagnostic messages may be included for supportability, but they are never the
source of truth.

Archive responses should report `pending` when environment archive/delete has
completed but the branch deletion operation has only been requested, queued, or
dispatched. `pending` is derived only from the active lifecycle operation row.

Audit candidates without branch ref rows are not `BranchCleanupReport` values.
Use a separate `BranchAuditCandidateReport` for discovered local branches that
have no persisted branch ref:

- `{ kind: "discovered_branch", candidateId, hostId, sourceGitCommonDir,
branchName, observedHeadSha, expectedHeadSha, checkoutStatus, inferredThreadId?,
auditConfidence?, ownership: "unknown", blockers }`

`candidateId` is non-persisted display identity derived from
`hostId + sourceGitCommonDir + branchName`. `expectedHeadSha` is the observed
head the user reviewed and must be round-tripped by any later review/claim or
manual cleanup request. Discovered candidates may appear in audit output and
may be selected for an explicit review/claim flow, but they are never eligible
for automatic cleanup until they become a persisted branch ref with
`bb_created` ownership.

## Server And Daemon Ownership

Server responsibilities:

- Decide whether a branch is BB-owned and eligible for automatic cleanup.
- Decide automatic vs manual behavior.
- Look up project sources, source git common dirs, environment records, thread
  liveness, archive state, and cleanup mode.
- Supply required policy fields to daemon commands.
- Queue durable `delete_branch` operations.
- Reconcile lost daemon results, expired commands, reconnects, and repeated
  cleanup requests.
- Format API responses and user-facing cleanup reports from the canonical wire
  contract.

Daemon responsibilities:

- Execute host-local git operations.
- Inspect worktree checkout state with `git worktree list --porcelain`.
- Resolve branch refs and expected SHAs.
- Delete local branches with safe or force git behavior.
- Return structured result and blocker codes without applying product policy.
- Do not accept DB row ids, environment ids, or ownership claims as daemon
  authority. The daemon validates git facts only.

Ownership correlation remains server-owned:

- Before queueing branch cleanup, the server verifies the active branch ref row,
  environment, host, branch name, ownership, and expected head.
- For manual apply, "expected head" means the client-supplied `expectedHeadSha`
  from a prior dry-run or audit response. For automatic apply, it means a head
  refreshed by the server in the same lifecycle transition that queues the
  daemon command.
- The daemon validates only git facts under lock: source git common dir, branch
  name, expected head SHA, checked-out state, protected refs, and merge proof.
- Before applying a successful daemon result, the server re-reads the branch ref
  row and environment state. If ownership, branch name, source common dir, or
  expected head no longer match, the server records `head_changed` or
  `shared_owner` rather than marking the branch deleted.

Add a daemon command such as `workspace.delete_branch` rather than hiding branch
deletion inside `environment.destroy`. A separate command supports dry runs,
manual cleanup, retries, and clear result reporting after the worktree path has
already been removed.

The daemon command must operate on the source repository git common dir, not the
managed worktree path. After `removeWorktree()` succeeds, the worktree path may
not exist, and branch state plus worktree checkout state live in the source
repository's shared git metadata.

Required daemon command input:

- `sourceGitCommonDir`: absolute source repo git common dir.
- `branchName`.
- `expectedHeadSha`.
- `protectedBranches`: server-supplied list or pattern payload with length at
  least 1.
- `mode`: `safe` or `force`.
- `operation`: `dry_run` or `apply`.
- Safe mode also requires `mergeBaseBranch`.
- Force mode must explicitly pass `mergeBaseBranch: null` to state that merge
  proof is not used for that operation.

The daemon should not default missing policy fields.

## Request Contracts

Avoid optional fields that hide defaults. Use discriminated unions with required
fields.

Add JSDoc for non-obvious contract types and route actions:

- `workspace.delete_branch` command schema must document source git common dir
  usage, lock/freshness semantics, `expectedHeadSha`, effective-merge meaning,
  force-mode limits, stale worktree handling, and result/blocker code meaning.
- `cleanup_branch` environment action schema must document that force branch
  cleanup is separate from force archive, that the confirmation literal is an
  intent sentinel, and that branch freshness is enforced by daemon-side
  expected-head checks.

Validation for these contracts should be concrete:

- Schema tests assert required fields, discriminants, absence semantics, enum
  values, and rejection of missing policy inputs.
- A doc lint or review checklist verifies JSDoc exists on the non-obvious
  contract exports. Do not rely on behavioral tests to prove prose quality.

Daemon command shape:

- Safe dry run/apply:
  - `type: "workspace.delete_branch"`
  - `operation: "dry_run"` or `"apply"`
  - `mode: "safe"`
  - `sourceGitCommonDir`
  - `branchName`
  - `expectedHeadSha`
  - `mergeBaseBranch`
  - `protectedBranches`
- Force apply:
  - `type: "workspace.delete_branch"`
  - `operation: "apply"`
  - `mode: "force"`
  - `sourceGitCommonDir`
  - `branchName`
  - `expectedHeadSha`
  - `mergeBaseBranch: null`
  - `protectedBranches`

The daemon schema should reject empty `protectedBranches` and should reject
`eligible` as an apply result.

Server action request shape:

- Extend `environmentActionRequestSchema` and
  `POST /environments/:id/actions` with `action: "cleanup_branch"`.
- Use an `options` discriminated union:
  - `{ operation: "dry_run", mode: "safe" }`
  - `{ operation: "apply", mode: "safe", expectedHeadSha }`
  - `{ operation: "apply", mode: "force", expectedHeadSha, confirmation:
"delete_branch" }`
- `confirmation: "delete_branch"` is an intent sentinel only. It proves the user
  intentionally selected force branch cleanup, but it is not a freshness token.
  Freshness is enforced by `expectedHeadSha` inside the daemon's locked apply
  section.
- Dry-run and audit responses must return the `expectedHeadSha` the user is
  reviewing. Manual apply requests must round-trip that exact value.
- For automatic cleanup with no client request, the server must refresh the
  branch head from the daemon immediately before creating the `delete_branch`
  operation and use the refreshed head as `expectedHeadSha`. The server must not
  use a stale `lastObservedHeadSha` as an apply freshness token unless it was
  refreshed in the same request or lifecycle transition.
- Do not add accepted-but-ignored fields. If a field exists in the route
  contract, implement it end to end.

Audit request shape:

- Use required scope fields, for example:
  - `{ scope: "environment", environmentId }`
  - `{ scope: "project", projectId, hostSelection }`
  - `{ scope: "all", hostSelection }`
- `hostSelection` should be explicit, such as `{ kind: "all" }` or
  `{ kind: "only", hostIds: [...] }`.
- Authorization must scale with audit scope:
  - `environment` scope uses the existing environment action authorization.
  - `project` scope requires authorization for the target project.
  - `all` scope requires an instance-admin/operator tier. If BB does not have
    that tier at implementation time, keep `scope: "all"` limited to trusted
    local admin tooling and do not expose it through regular project APIs.
- Apply remains environment-scoped even when a broad audit discovered the
  candidate. Broad audit results should link to explicit environment or branch
  cleanup requests rather than applying deletes directly.
- Dry run is explicit in request payloads; CLI can default to dry-run before
  building the payload.

## TOCTOU And Git Locking

Branch cleanup must avoid time-of-check/time-of-use races.

- Dry run is advisory only. Apply recomputes all checks.
- In apply mode, the daemon must acquire the worktree metadata lock for the
  source repo common dir before reading worktree checkout state, resolving the
  branch head, checking `expectedHeadSha`, evaluating effective merge, and
  deleting the branch.
- The expected-head check must happen inside the same locked section as the
  branch deletion.
- Before returning `checked_out_elsewhere`, the daemon should handle stale git
  worktree metadata under the same lock by running `git worktree prune` or by
  classifying worktree entries whose paths no longer exist as stale. Stale
  entries must not block branch deletion forever.
- If the branch head differs from `expectedHeadSha`, return terminal
  `BranchCleanupApplyResult` or `BranchCleanupDryRunResult`
  `{ code: "blocked", blockerCode: "head_changed" }` for the current command
  operation.
- If lock acquisition fails or git metadata cannot be read, return structured
  failure or blocker data; do not fall back to unlocked deletion.

## Edge Case Handling

Unmerged commits:

- Safe cleanup blocks branch deletion with `not_effectively_merged`.
- Force environment cleanup preserves the branch by default.
- Explicit force branch cleanup can delete only BB-owned branches and requires
  explicit confirmation in API/CLI/UI.

Dirty worktrees:

- Safe archive already blocks workspace cleanup.
- Force archive may remove the workspace, but branch deletion remains blocked
  unless the branch is otherwise safe or explicitly force-cleaned.

Active or archived threads:

- Live non-archived threads block environment cleanup and branch deletion.
- Archived threads do not block environment cleanup.
- Branch deletion should also check for any live environment sharing the same
  owned branch, not just the current environment.

Multiple environments sharing one branch:

- Because branch refs are per-environment ownership claims, multiple active
  rows may point at the same `(hostId, sourceGitCommonDir, branchName)`. Treat
  that as `shared_owner` unless all sibling branch refs for that branch identity
  are destroyed or deleted.
- Report the blocking environment ids in dry-run output.
- Re-run the server-side active-owner check immediately before queuing apply and
  again before marking a daemon result as deleted. The daemon re-checks only
  git facts under the worktree metadata lock; DB ownership remains server-side.

Branch checked out elsewhere:

- Block deletion with `checked_out_elsewhere` if
  `git worktree list --porcelain` shows the branch checked out in any worktree.
- This covers primary checkouts and user-created worktrees.

Protected/default branches:

- Never delete the default branch.
- Never delete the selected merge-base branch.
- Start with server-supplied protected names and patterns. Recommended initial
  defaults: `main`, `master`, `develop`, `trunk`, and release branches.
- Because automatic deletion is restricted to BB-owned branches, protected
  branch handling should be a last-resort guard, not the main safety mechanism.

User-created branches:

- Never auto-delete.
- Include them only in audit output when requested with an explicit flag.
- Require branch name selection for manual deletion.

Missing branch:

- Treat as idempotent success for deletion operations.
- Mark the branch ref deleted with result code `already_missing`.
- If the branch appears to have been renamed out of band, automatic ownership of
  the old branch name is abandoned. Mark the old ref `already_missing`; any new
  BB-looking branch discovered by audit is an `unknown` audit candidate until
  explicitly selected by the user.

Source repo unavailable:

- Retryable causes:
  - host disconnected
  - command timeout or transient git lock failure
  - source path temporarily inaccessible but the host cannot prove it is gone
- Terminal causes:
  - connected host confirms `sourceGitCommonDir` no longer exists
  - connected host confirms the path exists but is no longer a git common dir
  - project source was changed and no current source maps to the stored common
    dir
- Retryable causes keep the operation requested and retry on reconnect/sweep
  with blocker `source_repo_unavailable`.
- Terminal causes stop automatic retries and surface a terminal blocker. The
  user can repair the project source, rerun audit, or abandon the branch cleanup
  record.

Host disconnected:

- Keep operation requested.
- Periodic sweeps should retry when the host reconnects.
- Audit output should include `host_disconnected` per host instead of silently
  omitting that host's branches.

Lost or expired daemon result:

- Follow the existing lifecycle operation pattern used by provision/destroy.
- Expired branch deletion commands fail the branch deletion operation without
  changing environment status.
- Repeated requests should upsert or reuse the active branch deletion operation.

Force and safe precedence:

- Mirror `resolveRequestedCleanupMode()` semantics for branch cleanup requests:
  once an active branch cleanup operation has force intent, later safe requests
  cannot downgrade it.
- Safe requests may reuse an active safe operation.
- Force branch cleanup still refuses protected and checked-out branches.

State transitions:

| Current branch cleanup operation                        | Incoming request    | Behavior                                                                                                                                                                                                                                     |
| ------------------------------------------------------- | ------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| None                                                    | Safe apply          | Create safe `delete_branch` operation with the supplied/refreshed `expectedHeadSha`.                                                                                                                                                         |
| None                                                    | Force apply         | Create force `delete_branch` operation after confirmation, ownership, protected-branch, checked-out, and expected-head checks.                                                                                                               |
| Safe requested or queued, daemon command not dispatched | Safe apply          | Reuse the active operation and return its current report.                                                                                                                                                                                    |
| Safe requested or queued, daemon command not dispatched | Force apply         | Upgrade the active operation to force after confirmation and server-side revalidation; do not create a parallel operation.                                                                                                                   |
| Safe command dispatched                                 | Safe apply          | Reuse the active operation.                                                                                                                                                                                                                  |
| Safe command dispatched                                 | Force apply         | Record force escalation intent on the operation. If safe apply deletes or finds the branch missing, finish normally; if safe blocks only on merge proof, revalidate and dispatch a force command with the force request's `expectedHeadSha`. |
| Force requested, queued, or dispatched                  | Safe apply          | Reuse the force operation and report that safe cannot downgrade force intent.                                                                                                                                                                |
| Force requested, queued, or dispatched                  | Force apply         | Reuse the active force operation unless the supplied `expectedHeadSha` differs, in which case return or record `head_changed`.                                                                                                               |
| Terminal deleted or already missing                     | Safe or force apply | Return the terminal report idempotently.                                                                                                                                                                                                     |
| Terminal blocked or failed                              | Safe or force apply | Create a new retry operation only after rerunning server-side ownership, sharing, protection, and expected-head checks.                                                                                                                      |

Force confirmation, replay, and idempotency:

- `confirmation: "delete_branch"` is an intent sentinel, not authorization.
  Authorization follows the existing environment action authorization model.
- The sentinel is not freshness protection. Replayed or duplicated requests
  still need current server-side ownership checks and daemon-side
  `expectedHeadSha` validation under lock.
- Idempotency is owned by lifecycle operation identity. Repeated force apply
  requests should reuse or advance the existing operation rather than creating
  parallel branch deletion commands.

## Async Lifecycle Wiring

Implement branch cleanup as a server-owned lifecycle, not as extra states on
`environment.status`.

Required wiring:

- Extend domain operation kinds with `delete_branch`.
- Add branch cleanup request schemas for safe dry-run, plus
  `environment_operations` payload schemas for safe apply and force apply.
- Add host daemon command and result schemas for `workspace.delete_branch`.
- Register command result ownership in
  `apps/server/src/internal/command-result-owners.ts`.
- Extend lifecycle failure handling and the `LifecycleFailureReport` union in
  `apps/server/src/services/hosts/expired-commands.ts`.
  Add a variant shaped like
  `{ kind: "delete_branch", operationId, environmentId, branchRefId,
hostId, sourceGitCommonDir, branchName, failureCode, blockerCode?,
message? }`.
- Add completion/failure handlers that write terminal `BranchCleanupApplyResult`
  payloads to the owning `environment_operations` row. Successful deletion or
  already-missing results may also set `environment_branch_refs.deletedAt`, but
  branch refs do not store cleanup result or pending lifecycle state.
- Add command-result reconciliation that can repair a successful provisioning
  result whose environment was marked ready before its branch ref row was
  written.
- Add a periodic sweep wired through
  `apps/server/src/services/system/periodic-sweeps.ts` that advances requested
  branch cleanup operations when hosts reconnect.
- Add a ready-environment repair sweep for managed worktree environments that
  are missing active branch ref rows after provisioning success.
- Add command expiry behavior that marks the branch cleanup operation failed and
  preserves the branch ref.
- Add repeated-request handling that upserts or reuses the active operation.
- Add reconnect reconciliation so pending branch cleanup is retried after daemon
  reconnect when its host is available.
- Add server-start reconciliation for nonterminal `delete_branch` operations so
  a server restart re-claims requested, queued, and dispatched work the same way
  reconnect and expired-command paths do.
- Ensure the operation row is the only durable source for requested, queued,
  in-progress, failed, and completed branch cleanup state.

Observability:

- Emit structured logs for branch cleanup request, queue, dispatch, completion,
  retry, blocker, and sweep repair events with `operationId`, `environmentId`,
  `branchRefId`, `hostId`, `sourceGitCommonDir`, `branchName`, `mode`,
  `operation`, `result.code`, and `blockerCode` when present.
- Add counters for cleanup results and blocker codes, sweep batches processed,
  repair rows created, retry attempts, and command expirations.
- Add latency metrics for time from request to dispatch and request to terminal
  result. Phase 3 exit should include enough logging to debug branch leaks
  without manual database spelunking.

Exit criteria:

- Lost daemon results, expired commands, reconnects, and repeated requests all
  converge without changing `environment.status`.
- Failed branch deletion does not roll back successful workspace cleanup.

## Branch Ref Recovery And Reconciliation

Recovery rules live here so provisioning, command-result ownership, and sweeps
do not each invent partial behavior.

Provisioning success with branch-ref write failure:

- The provisioning success handler should write the branch ref in the same DB
  transaction that marks the environment `ready` when possible.
- If command-result side effects fail after the daemon has successfully
  provisioned the branch, command-result reconciliation should retry the missing
  branch ref write when it sees a successful provisioning result for a managed
  worktree.
- A ready-environment repair sweep should find `ready` managed worktree
  environments with no active branch ref row and reconstruct the row from the
  environment record plus daemon-observed branch/head/common-dir data.

Environment destroy success with branch-cleanup enqueue failure:

- Environment cleanup should create or reuse the `delete_branch` operation in
  the same server-side transition that records successful workspace destroy
  when possible.
- If workspace destroy succeeds but creating or queueing the branch cleanup
  operation fails, do not roll back the destroyed workspace. Preserve the active
  branch ref and leave cleanup recoverable through the lifecycle sweep.
- A destroyed-environment repair sweep should find destroyed managed worktree
  environments with active BB-owned branch refs, no terminal branch cleanup
  operation, and no active branch cleanup operation. It should re-run
  server-side eligibility checks, then enqueue cleanup or record a terminal
  blocker on the operation when automatic retry is not allowed.
- The repair sweep must not infer eligibility from branch name alone. It uses
  the persisted branch ref, environment record, source git common dir, and
  daemon-observed git facts.

Repair sweep policy:

- Run as part of the existing periodic sweep family in
  `apps/server/src/services/system/periodic-sweeps.ts`, with a small bounded
  batch size per tick to avoid expensive repo scans. Initial implementation
  should use explicit constants such as
  `BRANCH_REF_REPAIR_SWEEP_BATCH_SIZE`,
  `BRANCH_REF_REPAIR_SWEEP_INTERVAL_MS`,
  `DESTROYED_ENV_BRANCH_CLEANUP_SWEEP_BATCH_SIZE`, and
  `BRANCH_CLEANUP_AUDIT_REPO_CONCURRENCY`, and document the sweep cadence next
  to the lifecycle owner.
- Only query connected hosts.
- Use `bb_created` only when the environment and daemon observations prove the
  generated branch still matches the managed worktree.
- Create recovered rows as `unknown` when branch identity, source common dir,
  or head cannot be proven. Do not downgrade existing `bb_created` rows because
  of transient proof failures; record a blocker or retry instead.
- Never queue automatic branch deletion from a repaired row in the same sweep
  that created it. A later cleanup pass must re-evaluate eligibility.

This recovery belongs to Phase 3. Phase 1 only adds schema/contracts and
successful-path branch ref creation.

## UX, API, And CLI Plan

Status and reporting:

- `bb environment show <id>` should show branch ownership and canonical branch
  cleanup status: `not_applicable`, `audit_needed`, `not_attempted`,
  `pending`, `eligible`, `deleted`, `blocked`, or `failed`.
- `bb status` should show branch cleanup warnings only when the current thread's
  environment has a blocker that needs user action.
- Environment cleanup and archive results should include the derived archive
  outcome from `BranchCleanupReport`.
- The `eligible` result code means a dry-run branch cleanup would be safe.
  Archive outcome uses `skipped` for this case because archive did not delete
  the branch synchronously; if a deletion operation is queued, archive outcome
  should be `pending`.

API:

- Use `POST /environments/:id/actions` with discriminated
  `action: "cleanup_branch"` for environment branch cleanup.
- Use `POST /environments/actions` or another environment-scoped action route
  for project/global audits if an existing collection action pattern exists;
  otherwise add the smallest environment-scoped route that fits the contract.
- Return canonical `BranchCleanupReport` plus derived display/archive fields
  when the route already has those concepts.
- Audit routes return a list of branch-ref-backed `BranchCleanupReport` rows
  plus discovered `BranchAuditCandidateReport` rows. Keep those types distinct
  so scripts cannot treat an unowned discovered branch as an apply-ready branch
  ref.
- Display derives from branch-ref existence plus the active or latest
  `delete_branch` operation:
  - unmanaged environment, managed clone, or no local branch target:
    `not_applicable`
  - managed worktree with no branch ref row: `audit_needed`
  - branch ref row exists and no branch cleanup operation exists:
    `not_attempted`
  - active branch cleanup operation exists: `pending`
  - completed branch cleanup operation or dry-run response exists: derive from
    `BranchCleanupDryRunResult` or `BranchCleanupApplyResult`

CLI:

- Prefer existing `bb environment` grouping and positional subcommands, matching
  commands such as `bb environment commit <id>` and
  `bb environment squash-merge <id>`.
- Implement the CLI in `apps/cli/src/commands/environment.ts`, with shared
  formatting in existing CLI helper modules where appropriate.
- Add `bb environment cleanup-branch <environmentId> --dry-run`.
- Add `bb environment cleanup-branch <environmentId> --apply`.
- Add
  `bb environment cleanup-branch <environmentId> --apply --force --confirm delete_branch`.
- Add broader audit under the same group, for example
  `bb environment audit-branches --project <projectId> --dry-run`.
- Dry run can be the CLI default, but the API payload must still send an
  explicit dry-run operation.
- Human output should list branch name, source repo, host, owning
  environment/thread, merge status, checkout status, result code, and blocker.
- JSON output should expose the same structured fields for scripts.

Prompts and flags:

- Archive confirmation should keep focusing on workspace data loss.
- After squash merge, report that the branch remains checked out by the
  environment and that branch cleanup can happen only after the user archives
  or otherwise cleans up the environment.
- If branch deletion is clean and automatic, report it after the fact.
- If branch deletion is blocked, show the branch and typed blocker reason.
- Force archive should never imply force branch cleanup.
- Branch cleanup should never imply remote branch deletion.

## Migration Strategy

Do not bulk-delete existing polluted branches automatically.

1. Add the branch ownership model for newly created managed worktrees.
2. Backfill branch ref rows for existing non-destroyed managed worktree
   environments with ownership `unknown`.
3. Audit may compute non-persisted confidence from branch names, but confidence
   must not change ownership or automatic deletion eligibility.
4. Build an audit command that scans real repositories:
   - DB environments with `managed=true` and `workspaceProvisionType` of
     `managed-worktree`
   - persisted branch ref rows first
   - local branches parsed by a shared `parseManagedBranchName()` helper that
     is the inverse of `buildManagedBranchName()`
   - the anchored fallback pattern `^bb/(.*-)?(thr_[a-z0-9]+)$` only inside
     that shared parser as a paranoid audit candidate source, never as
     ownership proof
   - `git worktree list --porcelain` checkout state
   - active and archived thread references
   - merge status against default or stored merge-base branch
5. Default migration output to dry run.
6. Allow `--apply --merged-only` for branches with persisted `bb_created`
   ownership only.
7. Require `--force`, explicit branch/environment selection, and confirmation
   for unmerged BB-owned branches.
8. Keep unknown-ownership branches opt-in and manual; never include them in
   automatic cleanup.
9. Branches discovered by audit with no matching environment record are
   audit-only candidates with ownership `unknown`. Safe cleanup requires
   explicit user selection plus merged-only proof; force cleanup requires
   explicit branch selection and force confirmation.
10. After one release cycle, consider showing non-blocking warnings for old
    polluted branches, but keep deletion opt-in.

## Multi-Host Audit Behavior

Audits must not assume a project has one local source or one connected host.

- Project audit iterates project sources by host.
- Global audit iterates branch ref rows grouped by host and source git common
  dir, then separately scans connected source repositories for discovered
  branches that become `BranchAuditCandidateReport` rows.
- For each connected host, the server sends daemon audit/dry-run commands for
  that host's source repositories.
- For disconnected hosts, audit returns per-host rows with
  `blockerCode: "host_disconnected"` and does not mark branches eligible.
- Results are merged by canonical branch ref id when present, otherwise by
  `BranchAuditCandidateReport.candidateId` for display only.
- Cross-host branch names are not the same branch. Host id and source git common
  dir are part of audit context, but not a natural primary key.
- CLI output should group audit results by host and source repository.

## Implementation Phases

Phase dependencies and boundaries:

- Phase 1 is required before any automatic deletion. Without persisted
  ownership, cleanup must stay audit-only.
- Phase 2 can be built and tested independently through daemon/host-workspace
  tests.
- Phase 3 can land with durable lifecycle wiring and minimal JSON/log
  visibility, but automatic deletion must be limited to environments with
  active branch refs created after Phase 1 provisioning or with refs created by
  Phase 5 backfill/review. Existing managed environments without branch refs
  must return an audit-needed report, not `not_applicable`, until Phase 5 has
  run.
- Broad user rollout depends on Phase 4 so users can understand pending,
  blocked, and failed branch cleanup.
- Phase 4 depends on the canonical wire contract from Phase 1 and the lifecycle
  states from Phase 3.
- Phase 4 owns user-facing environment cleanup/audit commands and API shapes.
- Phase 5 owns migration/backfill policy and rollout guidance; the Migration
  Strategy section is the source of truth for those rules. Phase 5 depends on
  the Phase 1 ownership schema plus Phase 4 audit/CLI surfaces.

### Phase 1: Contract And Ownership Tracking

Goal: Record branch ownership for new managed worktrees.

Changes:

- Add branch ownership domain types and canonical `BranchCleanupReport`,
  `BranchCleanupDryRunResult`, and `BranchCleanupApplyResult` contract types.
- Add DB storage for branch refs with generated id PK and environment FK
  `ON DELETE CASCADE`.
- Extend successful provisioning result handling to create branch ref rows for
  managed worktrees after the daemon confirms branch/head/common-dir data.
- Write the branch ref row atomically with marking the environment ready where
  possible. Reconciliation and repair sweeps are Phase 3 work.
- Update provisioning result handling to fill `initialHeadSha`,
  `lastObservedHeadSha`, and `sourceGitCommonDir`.
- Add query helpers for branch refs by id, environment, host/source common dir,
  and branch name.
- Add provisioning branch collision checks so `git worktree add -B` never
  resets unknown or user-owned refs silently.

Exit criteria:

- New managed worktree environments have a branch ref with ownership
  `bb_created`.
- Failed provisioning does not create a BB-owned branch ref row.
- Unmanaged environments have no auto-owned branch ref.
- Provisioning fails or chooses a new generated branch when the target branch
  already exists with unknown, user-owned, or absent ownership.
- Same-environment reprovision reuses/resets a branch only after server-side
  and daemon-side locked collision checks both prove it is the same BB-owned
  claim.
- No automatic branch deletion, branch cleanup lifecycle, or recovery sweep
  behavior changes yet.

### Phase 2: Daemon Branch Deletion Primitive

Goal: Add a host-local, policy-free git primitive.

Changes:

- Add `workspace.delete_branch` to the host daemon contract.
- Implement dry-run and apply modes in `packages/host-workspace`.
- Use source repo git common dir as the command root.
- Recompute all checks in apply mode under the worktree metadata lock.
- Extract reusable effective-merge detection so squash-merged/no-net-change
  branches can be safely deleted.
- Return canonical `BranchCleanupDryRunResult` and `BranchCleanupApplyResult`
  payloads.

Exit criteria:

- Safe mode deletes merged and effectively merged local branches.
- Safe mode refuses unmerged, protected, moved, and checked-out branches.
- Force mode still refuses protected and checked-out branches.
- Expected-head checks happen under the same lock as deletion.
- Dry run never mutates refs or DB-observed-head fields.

### Phase 3: Server-Owned Branch Cleanup Lifecycle

Goal: Queue and reconcile branch cleanup as durable lifecycle work.

Changes:

- Add `delete_branch` environment operation kind.
- Queue branch deletion only after environment destroy succeeds.
- Register command result ownership, failure handling, expiry handling, and
  periodic sweeps.
- Add branch ref recovery/reconciliation and ready-environment repair sweeps.
- Record branch deletion completion, blockers, and failures as typed terminal
  payloads on `environment_operations`.
- Keep force environment cleanup from auto-deleting unmerged branches.
- Add force/safe precedence resolution for repeated requests.

Exit criteria:

- Safe archive of a clean, effectively merged managed worktree removes both
  worktree and branch.
- Existing managed worktrees that predate branch refs are preserved and reported
  as audit-needed until Phase 5 backfill/review creates a branch ref; Phase 3
  must not silently treat them as `not_applicable`.
- Force archive removes the worktree and preserves an unmerged branch with a
  visible cleanup blocker.
- Lost or expired daemon branch deletion commands are retried or failed
  durably without changing environment status.
- Repeated cleanup requests are idempotent against the operation identity.
- If workspace destroy succeeds but branch cleanup operation creation or enqueue
  fails, the repair sweep later creates or advances the missing operation
  without resurrecting the workspace.

### Phase 4: UX, API, And CLI

Goal: Make branch cleanup visible and controllable.

Changes:

- Extend environment actions with cleanup branch dry-run/apply requests.
- Add project/global audit API with explicit multi-host behavior.
- Add `bb environment cleanup-branch` and `bb environment audit-branches` CLI
  commands in `apps/cli/src/commands/environment.ts`.
- Update `bb environment show`, `bb status`, and archive reporting.
- Add UI copy only where user confirmation is required.

Exit criteria:

- Users can see why a branch was deleted, skipped, or blocked.
- Audit commands default to dry run at the CLI and send explicit dry-run API
  payloads.
- Force branch deletion requires explicit user intent.
- CLI/API/UI use the canonical wire contract.

### Phase 5: Migration And Audit

Goal: Address existing polluted branches safely.

Changes:

- Implement the Migration Strategy section as the source of truth for backfill,
  no-environment audit candidates, branch-name parsing, and apply/force
  cleanup behavior.
- Keep Phase 5 changes limited to migration/backfill code, audit command
  rollout, and operator documentation for the recommended cleanup flow.

Exit criteria:

- Existing polluted branches can be found and cleaned without manual SQL.
- No existing branch is deleted during migration without an explicit apply.
- Unknown-ownership branches stay preserved unless explicitly selected.

- `--force --apply` requires explicit selection or environment id plus
  confirmation.
- Multi-host audit reports disconnected hosts and groups connected-host
  results by host/source repository.
- Audit candidates with no environment record are reported as unknown/audit-only
  `BranchAuditCandidateReport` rows and require explicit selection for review
  or claim before cleanup.
- Audit uses the shared `parseManagedBranchName()` helper rather than a
  route-local regex copy.
- Environment action requests reject missing required fields instead of
  defaulting them.
- Force cleanup rejects missing or incorrect confirmation sentinel, and tests
  assert the sentinel does not bypass `expectedHeadSha`.
- Schema tests assert required fields, discriminants, absence semantics, and
  wire enum values.
- Schema tests reject `eligible` in `BranchCleanupApplyResult`, reject empty
  `protectedBranches`, and assert CLI `--confirm delete_branch` maps directly
  to API `confirmation: "delete_branch"`.
- Doc lint or review checklist verifies required JSDoc exists for
  `workspace.delete_branch` and `cleanup_branch`.

Suggested commands:

- `pnpm exec turbo run test --filter=@bb/host-workspace`
- `pnpm exec turbo run test --filter=@bb/host-daemon`
- `pnpm exec turbo run test --filter=@bb/server`
- For slow suites, redirect output to a file first, then inspect the file.

## Deferred Product Decisions

These are not required before the first implementation:

- Remote branches: keep remote branch deletion out of scope unless explicitly
  added as a separate feature.
- Protected branch configuration: start with default branch plus common
  trunk/release names; add project-level protected patterns only if users need
  custom protection.
- Migration confidence display: keep confidence non-persisted. Add a persisted
  confidence/ownership enum only if it becomes a reviewed product state.

## Rollup Acceptance Gate

The implementation is complete only when the phase-specific exit criteria above
pass and the following end-to-end outcomes are true:

- New managed worktree cleanup no longer leaves safe BB-owned local branches
  behind.
- User-created and unknown-ownership branches are preserved automatically.
- Force cleanup does not silently remove the last ref for unmerged work.
- Branch cleanup is visible in API, CLI, and UI reporting through one canonical
  wire contract.
- Existing polluted branches can be audited and cleaned with dry-run and apply
  commands.
- Tests exercise real git repos/worktrees and real SQLite migrations.
- Async lifecycle behavior covers lost daemon results, expired commands,
  reconnects, and repeated requests.
