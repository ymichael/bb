# Host Daemon Event Protocol Hard Cutover

## Current Status (2026-05-04)

Open future work. The short-term timeline safety fixes have landed separately in
this branch: server-side sequence-conflict detection around the current
`INSERT OR IGNORE` path, daemon 409/rebase retry handling, canonical duplicate
comparison, and bounded repeated rebase retries.

Those fixes prevent the known corruption class in the current protocol, but they
do not implement this hard cutover. This plan remains the follow-up work to make
the server own final event sequencing and then delete the transitional
conflict/rebase path.

## Motivation

The timeline failure on `thr_bj3p5vk9py` exposed a bad ownership boundary in the current event protocol.

The server and host daemon can both choose final per-thread `sequence` values. In the observed failure, the server persisted a `client/turn/requested` event at sequence `7340` while the daemon had already buffered a provider `turn/started` event with the same thread sequence. At the storage layer, current event ingestion still relies on `INSERT OR IGNORE`; before the short-term fix, that silently kept the server event and dropped the daemon `turn/started`, then accepted later daemon events for the same turn. Projection then saw `turn/input/accepted` without the required `turn/started` and refused to build the timeline.

The immediate conflict/rebase fix layers collision detection, a `409` response, and daemon rebase behavior around those current `INSERT OR IGNORE` semantics. It prevents silent corruption in the current protocol, but it still keeps the daemon in the business of guessing final log positions. The proper fix is to make the event log append-only under server ownership:

- the daemon produces events with stable producer identity, not final log positions;
- the server assigns canonical per-thread sequence numbers transactionally;
- retries are deduplicated by event identity, not by caller-guessed sequence;
- user request correlation uses a durable request id, not `clientRequestSequence`.

## Current Context

Today, daemon events cross the boundary as `HostDaemonEventEnvelope` records with a daemon-assigned `sequence`. The server stores events with a uniqueness constraint on `(threadId, sequence)`, and projection treats `sequence` as the canonical thread log order.

Several code paths also use `clientRequestSequence` as a correlation handle:

- `client/turn/requested` is persisted at a server-assigned sequence;
- `thread.start` and `turn.submit` carry that sequence to the daemon;
- `environment.provision.initiator.eventSequence` carries a sequence to seed
  streamed provisioning progress events;
- runtime/provider translation emits `turn/input/accepted.clientRequestSequence`;
- provider-translated `threadEventItem.userMessage.clientRequestSequence`
  records the same correlation on accepted user-message items;
- timeline projection matches the accepted input back to the requested input by sequence.

That coupling makes `sequence` serve two different purposes: final log order and client request identity. The hard cutover separates those concerns.

## Target Protocol

Protocol version: bump the host daemon protocol again and hard require the new shape on both sides. Do not keep compatibility branches for old daemons in application code.

This is a dev-mode hard cutover. The dev server and host daemon must be restarted together after the migration. If the server and daemon are upgraded out of order, session open/protocol-version validation must fail and the mismatched processes must refuse to talk instead of translating old and new event envelopes.

Daemon-to-server event envelope:

```ts
type HostDaemonEventEnvelope = {
  producerEventId: HostDaemonProducerEventId;
  threadId: ThreadId;
  event: ThreadEvent;
};
```

Server response for accepted event batches:

```ts
type HostDaemonEventBatchAcceptedResponse = {
  acceptedEvents: Array<{
    producerEventId: HostDaemonProducerEventId;
    threadId: ThreadId;
    sequence: number;
  }>;
};
```

Client turn request correlation:

```ts
type ClientTurnRequestedEvent = {
  type: "client/turn/requested";
  requestId: ClientTurnRequestId;
  // existing explicit target/prompt/options fields stay as-is
};

type TurnSubmitCommand = {
  type: "turn.submit";
  requestId: ClientTurnRequestId;
  // no eventSequence
};

type ThreadStartCommand = {
  type: "thread.start";
  requestId: ClientTurnRequestId;
  // no eventSequence
};

type TurnInputAcceptedEvent = {
  type: "turn/input/accepted";
  clientRequestId: ClientTurnRequestId;
  // no clientRequestSequence
};
```

Use repo-style prefixed ids rather than plain UUID strings:

- `producerEventId`: `hdevt_<id>`, generated once when an event enters the daemon buffer.
- `clientRequestId`: `creq_<id>`, generated by the server thread-event owner
  before persisting `client/turn/requested`.

Use a longer suffix than the existing short pretty ids for `hdevt_*` because daemon events are high volume. The id must be opaque: no session id, stream id, command id, or sequence encoded into it.

## Design Rules

- `sequence` is server-assigned final log position only.
- The daemon never sends `sequence`.
- The daemon never rebases events.
- Event buffer acknowledgement is by `producerEventId`.
- The server deduplicates daemon retries by `producerEventId`.
- Same `producerEventId` plus same payload is accepted idempotently and returns the original assigned sequence.
- Same `producerEventId` plus different payload is a hard protocol/data-integrity error.
- Payload equality uses one shared canonicalization helper for the daemon spool
  and server duplicate detection: validate through the domain schema, serialize
  with stable object-key ordering and no non-semantic whitespace, then hash the
  canonical payload with its event type, thread id, and protocol version.
- `clientRequestId` replaces `clientRequestSequence` everywhere in runtime, daemon commands, persisted events, and projection.
- `threadEventItem.userMessage.clientRequestId` replaces
  `threadEventItem.userMessage.clientRequestSequence` for provider user-message
  item correlation. The field remains optional only where provider events do not
  correspond to a server-created client turn request. Absence means there is no
  server request behind that provider user-message item; if a server-created
  request exists, missing `clientRequestId` is a protocol/projection error, not
  tolerated lost correlation.
- Session ids, stream ids, and command ids remain metadata for their own domains. They are not idempotency keys.
- The daemon event buffer is host-local transport durability, not product
  lifecycle state. The daemon owns persisting unsent and unacknowledged outbound
  event records; the server owns final event ordering and idempotent ingestion.

## Implementation Plan

### Phase 1: Contract Types

- Add `HostDaemonProducerEventId` and `ClientTurnRequestId` shared domain/contract types.
- Change `/session/events` request schema to require `producerEventId` and reject daemon-supplied `sequence`.
- Change `/session/events` success response to return only `acceptedEvents`.
- Remove `acceptedSequences` and sequence-conflict response handling from the contract. The new protocol has no legal sequence conflict class.
- Change `thread.start` and `turn.submit` to require `requestId` and remove
  `eventSequence`.
- Change `environment.provision.initiator` to remove `eventSequence`; the
  initiator keeps `threadId` and `provisioningId` only. Provisioning progress
  events go through the same producer-event-id buffer path as other daemon
  events.
- Change `turn/input/accepted` schemas to require `clientRequestId` and remove `clientRequestSequence`.
- Change `threadEventItem.userMessage` schemas to use optional
  `clientRequestId` and reject `clientRequestSequence`.
- Bump `HOST_DAEMON_PROTOCOL_VERSION`.

Exit criteria:

- `rg "clientRequestSequence|eventSequence|acceptedSequences|sequence_conflict" packages/host-daemon-contract packages/domain` returns no production matches.
- Contract tests cover the new event batch request/response, prove
  `threadHighWaterMarks` is not part of the `/session/events` success response,
  and reject the old sequence-bearing request shape.

### Phase 2: Server Append Ownership

- Update the server internal event ingestion path so it allocates per-thread `sequence` values inside the database transaction.
- Insert daemon events in request order per batch.
- Add durable storage for `producerEventId` and the canonical payload identity
  hash.
- Add a unique index on `producerEventId` for daemon-produced events.
- On retry:
  - if `producerEventId` exists and payload matches, return the existing `(threadId, sequence)`;
  - if `producerEventId` exists and payload differs, reject the batch as a data-integrity error;
  - if it does not exist, append the event with the next server-owned sequence.
- Keep server-created events on the same server-owned sequence allocator.

Exit criteria:

- No server ingestion code calls `INSERT OR IGNORE` for thread events.
- Tests prove that duplicate retry batches do not append duplicate events.
- Tests prove that a repeated `producerEventId` with a different payload is rejected.
- Tests prove semantically identical canonical payloads produce the same equality
  hash and payloads differing after canonicalization are rejected.
- Tests prove that mixed new and retried events return correct sequence mappings.

### Phase 3: Host Daemon Buffer

- Replace the in-memory-only event buffer with a daemon-owned durable spool under
  the host daemon data dir.
- Implement the spool as a small local SQLite database at
  `<dataDir>/event-spool.sqlite`. This is host-local transport durability, not a
  server DB migration.
- Use an explicit schema version and one ordered outbound table:
  - `localOrder INTEGER PRIMARY KEY AUTOINCREMENT`;
  - `producerEventId TEXT NOT NULL UNIQUE`;
  - `threadId TEXT NOT NULL`;
  - `eventType TEXT NOT NULL`;
  - `payloadJson TEXT NOT NULL`;
  - `payloadHash TEXT NOT NULL`;
  - `createdAt TEXT NOT NULL`;
  - `lastPostAttemptAt TEXT`;
  - `postAttemptCount INTEGER NOT NULL DEFAULT 0`.
- Open the spool with durable SQLite settings, including `PRAGMA journal_mode=WAL`
  and `PRAGMA synchronous=FULL`. Each `push()` inserts in a transaction
  that commits before `push()` returns, so the record is durable before it can be
  included in a POST body.
- Serialize concurrent `push()` operations through one in-process writer queue.
  `localOrder` defines resend order across concurrent pushes and process
  restart; flush reads unacknowledged records by ascending `localOrder`.
- Generate `producerEventId` once in `createEventBuffer.push()` and persist the
  full outbound record before it can be included in a POST body.
- On POST attempt, update `lastPostAttemptAt` and `postAttemptCount` in the same
  spool transaction that snapshots the batch to send.
- Reload the durable spool on daemon process start before opening a new session
  or fetching commands.
- Startup creates an empty spool if no spool exists. If a spool exists, run schema
  version validation and `PRAGMA integrity_check` before sending any event or
  fetching commands.
- No spool data migration is needed for the first rollout because the current
  buffer is in memory only and the upgrade runbook stops the daemon before the
  protocol bump. Future spool schema changes require explicit versioned
  migrations; unknown schema versions fail closed.
- Stop assigning or storing daemon-side `sequence`.
- Stop computing high-water based next sequence values.
- Stop rebasing buffered events.
- On successful POST, drop buffered events by returned `producerEventId`.
- Delete acknowledged rows in one transaction after validating the response
  contains only producer ids present in the sent snapshot. Acknowledgement for an
  already-deleted id is idempotent only if the payload was previously sent in the
  current process; otherwise log and fail closed as a protocol inconsistency.
- On reconnect or unknown POST outcome, resend the same buffered records with the same `producerEventId`.
- Treat local spool corruption as a fail-closed daemon startup/protocol error
  that requires explicit operator recovery; do not silently regenerate
  `producerEventId`s for records that may already have been committed by the
  server.
- The accepted loss window is only unrecoverable host-local storage loss before
  a record is durably written. Once `push()` returns, process restart,
  websocket reconnect, command-result retry, and unknown POST outcome must reuse
  the same `producerEventId`.
- Remove host-daemon `app.ts` wiring that seeds or rebases the event buffer:
  command-dispatch `seedThreadHighWaterMark`, command-result
  `eventBuffer.rebase()`, session-open `eventBuffer.seed()`, and any
  high-water-mark plumbing around `eventSink.emit()`/`flush()`.

Exit criteria:

- `apps/host-daemon/src/event-buffer.ts` has no sequence allocation or rebase logic.
- `apps/host-daemon/src/app.ts` has no event-buffer seed/rebase/high-water
  wiring.
- Buffer tests cover lost response/retry behavior and verify identical ids are reused.
- Buffer tests cover daemon process restart by persisting unacknowledged records,
  constructing a new buffer instance, and verifying the resent ids and payloads
  are identical.
- Buffer tests cover concurrent pushes preserving deterministic `localOrder`,
  ack deletion by `producerEventId`, spool schema mismatch, and corruption
  fail-closed behavior.
- Server client tests parse `acceptedEvents` only and no longer parse
  `threadHighWaterMarks` or sequence conflicts from event batch responses.

### Phase 4: Request Correlation Cutover

- Generate `clientRequestId` when the server appends `client/turn/requested`.
  The owner is `apps/server/src/services/threads/thread-events.ts`, specifically
  `appendClientTurnEvent` and `appendClientTurnEventInTransaction`; callers must
  not mint request ids independently.
- Persist that id in the event payload.
- Send the id through `turn.submit`.
- Thread command handlers pass `clientRequestId` into runtime.
- Runtime/provider adapters emit `turn/input/accepted.clientRequestId`.
- Runtime/provider adapters and event translators emit
  `threadEventItem.userMessage.clientRequestId` instead of
  `clientRequestSequence` when a provider user-message item is correlated to a
  server-created request.
- Timeline projection matches requests and accepted inputs by `clientRequestId`.
- Remove runtime and projection support for `clientRequestSequence`.

Exit criteria:

- `rg "clientRequestSequence|eventSequence" apps packages` returns no production matches.
- Timeline tests cover new-turn request acceptance and active-turn steer acceptance by `clientRequestId`.
- Existing `thr_bj3p5vk9py` timeline still loads after the one-off dev DB migration below.

### Phase 5: One-Off Dev DB Migration

This is a dev-only migration for `~/.bb-dev/bb.db`, not a product migration with backward compatibility guarantees.

Steps:

1. Stop the dev server and host daemon.
2. Run read-only preflight checks before any mutation:
   - find every `turn/input/accepted` event with `clientRequestSequence`;
   - require a matching `client/turn/requested` event in the same thread at that sequence;
   - find every stored `threadEventItem.userMessage.clientRequestSequence`
     inside `item/started` and `item/completed` payloads;
   - require a matching `client/turn/requested` event in the same thread at
     each referenced user-message sequence;
   - assert `(threadId, sequence)` uniqueness for all stored thread event rows;
   - separately assert every `client/turn/requested` row has a unique
     `(threadId, sequence)` before generating deterministic legacy
     `clientRequestId`s;
   - abort without mutation if any check fails, printing orphan
     `(threadId, sequence)` pairs and duplicate event rows.
3. Gate fetched/in-flight daemon commands before any mutation. The default is
   fail closed: if any fetched or in-flight old-protocol command cannot be
   unambiguously rekeyed to a `requestId`, stop and leave the DB untouched. An
   operator may explicitly choose a drain/wait mode only before mutation and only
   after verification shows there are no fetched/in-flight old-protocol rows
   remaining. Do not allow an old daemon to finish an old `eventSequence`
   command after the protocol version bump.
4. Back up the dev database to `/tmp/bb-dev-before-event-protocol-hard-cutover-<timestamp>.db`.
5. Add any new DB columns/indexes needed by the implementation, including daemon `producerEventId` storage.
6. Backfill existing daemon-produced events with deterministic legacy ids:

   ```text
   hdevt_legacy_<hash(threadId:sequence:eventType:eventPayloadHash)>
   ```

7. Backfill existing `client/turn/requested` events with deterministic request ids:

   ```text
   creq_legacy_<hash(threadId:sequence)>
   ```

8. Rewrite existing `turn/input/accepted.clientRequestSequence` fields to `clientRequestId` by looking up the `client/turn/requested` event with the same `(threadId, clientRequestSequence)`.
9. Rewrite existing `threadEventItem.userMessage.clientRequestSequence` fields
   inside `item/started` and `item/completed` payloads to `clientRequestId` by
   looking up the same-thread `client/turn/requested` event.
10. Rewrite any stored daemon commands or pending command payloads that still
    contain `eventSequence`: `turn.submit.eventSequence` ->
    `turn.submit.requestId`; `thread.start.eventSequence` ->
    `thread.start.requestId`; and
    `environment.provision.initiator.eventSequence` is removed, leaving
    `threadId` and `provisioningId`.

11. Remove `clientRequestSequence` and `eventSequence` from migrated event and
    command payloads.
12. Run an integrity check proving every `turn/input/accepted` has
    `clientRequestId`; every stored `threadEventItem.userMessage` with
    client-request correlation has `clientRequestId` and no
    `clientRequestSequence`; every `clientRequestId` references an existing
    `client/turn/requested` in the same thread; every daemon-produced event has
    a unique `producerEventId`; no stored daemon command payload contains
    `eventSequence`; and every thread has a dense, increasing server-owned
    sequence history.
13. Start the dev server and host daemon against the migrated DB.

Exit criteria:

- `thr_bj3p5vk9py` timeline endpoint returns `200`.
- SQL integrity checks return zero violations.
- No old protocol fields remain in event JSON or pending command JSON in the dev DB.

### Phase 6: Delete Legacy Code

- Delete sequence conflict/rebase code added as the safety fix for the old protocol.
- Delete old tests that assert daemon sequence conflict behavior.
- Delete old contract exports and types related to daemon sequence keys.
- Delete request-correlation fallback logic from thread-view.
- Delete any old DB helpers that only exist to look up sequence conflicts.

Exit criteria:

- `rg "sequence_conflict|acceptedSequences|clientRequestSequence|eventSequence|eventBuffer\\.rebase|eventBuffer\\.seed\\(|seedThreadHighWaterMark" apps packages` returns no production matches, except unrelated words or migration scripts.
- The protocol accepts only the new event envelope shape.
- The codebase has no runtime compatibility path for old daemon event sequences.

## Completion Criteria

The change is complete when:

- final event ordering is assigned only by the server;
- daemon event retries are idempotent by `producerEventId`;
- request/acceptance correlation uses `clientRequestId`;
- old daemon sequence protocol code has been deleted, not kept as a fallback;
- the one-off dev DB migration has been run and verified;
- Turbo typecheck and tests pass for all affected packages;
- the broken thread and a newly-created thread both load timelines successfully.
